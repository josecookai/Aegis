# Aegis: Mobile UX Specification

---

## 0. 界面速查（Quick Reference）

### 主导航（4 个 Tab）

| Tab | 用途 | 关键内容 |
|-----|------|----------|
| **首页** | 待审批列表 | Pending 请求卡片（Agent、金额、收款方、时间） |
| **历史** | 审计列表 | 所有请求（时间、状态、金额、收款方、Agent），可筛选 |
| **资产** | 钱包与卡管理 | ETH/SOL 钱包、信用卡（掩码），添加/删除/设为默认 |
| **设置** | 配置 | 通知开关、默认支付方式、生物识别、账户 |

### 关键界面

| 界面 | 用途 | 关键元素 |
|------|------|----------|
| **待审批列表** | 展示 pending 请求 | 卡片：Agent、金额、收款方、描述、时间；点击进入详情 |
| **审批详情** | 用户决策 | 大字号金额、收款方、描述、支付方式；批准/拒绝按钮；批准前 Face ID/Touch ID |
| **审批结果** | 反馈 | Toast 或结果页：成功/失败/已拒绝 |
| **添加资产** | 添加钱包/卡 | 加密货币：输入私钥（仅本地）；信用卡：表单（卡号、CVV 等，经 PCI 金库） |
| **历史/审计** | 查询记录 | 列表：时间、状态、金额、收款方；筛选与搜索；详情页展示原始请求与执行结果 |

### 推送与深链

- **推送文案：** `Aegis: 审批请求` + `您的 AI 助手请求向 {收款方} 支付 {金额}，请审批。`
- **Deep Link：** `aegis://approve?request_id={id}` 或 Universal Link
- **点击推送：** 打开 App → 解析 `request_id` → 拉取详情 → 展示审批页

**完整文档：** 见下方各章节

## 1. 设计原则

- **安全优先：** 所有敏感操作（批准支付、添加/导出凭证）必须经过生物识别（Face ID / Touch ID）或设备 PIN；敏感数据仅在安全芯片内处理，界面不展示完整私钥或 CVV。
- **信息透明：** 金额、币种、收款方、描述在审批前后始终清晰可见；审计历史完整可查，无隐藏操作。
- **操作极简：** 审批流程以「一键批准 / 一键拒绝」为核心；主要路径步骤最少，减少认知负担。

## 2. 信息架构（IA）

### 2.1. 主导航

| 模块 | 说明 |
|------|------|
| **首页（Home）** | 待审批列表 + 快捷入口；无待审批时展示空状态与「最近动态」摘要。 |
| **历史（History）** | 审计列表：所有请求的创建时间、状态、金额、收款方、来源 Agent；支持筛选与搜索。 |
| **资产（Assets）** | 已添加的加密货币钱包与信用卡；添加/删除/设为默认；仅展示掩码或地址，不展示私钥/CVV。 |
| **设置（Settings）** | 通知开关、默认支付方式、生物识别、账户与安全、关于与合规说明。 |

### 2.2. 导航形式

- 底部 Tab：首页、历史、资产、设置（或采用等效主导航模式）。
- 从推送进入时通过 Deep Link 直接到达「审批详情」页，完成后再返回首页或历史。

## 3. 关键界面与状态

### 3.1. 首页 / 待审批列表

**用途：** 展示当前用户所有 pending 的审批请求。

**元素：**

- 标题：「待审批」或「Approval Requests」。
- 列表：每条为一张卡片或行，包含：
  - 来源 Agent 名称或图标（若可展示）。
  - 金额 + 币种（主视觉）。
  - 收款方名称。
  - 简短描述（可选一行）。
  - 请求时间（相对或绝对）。
- 点击条目：进入「审批详情」页，并传入 `request_id`。
- 空状态：无待审批时展示文案如「暂无待审批请求」，可配插图。

**状态：** 列表支持下拉刷新，与后端同步 pending 列表。

---

### 3.2. 审批详情

**用途：** 展示单条请求的完整信息，用户在此做出批准或拒绝。

**元素：**

- **金额与币种：** 大字号、醒目（如 `$149.99 USD`）。
- **收款方：** 名称清晰展示。
- **描述：** 完整描述（如航班号、订单说明）。
- **支付方式：** 若请求中指定了 `payment_method_preference`，展示将使用的支付方式；否则展示「用户可选」或默认方式，并可切换（若有多张卡/多个钱包）。
- **主 CTA：** 
  - 「批准」：主按钮；点击后先触发 Face ID / Touch ID，通过后再提交批准。
  - 「拒绝」：次按钮或文字链；点击后确认一次再提交拒绝，可不强制生物识别（按产品策略）。
- **来源：** 可选展示「由 XXX Agent 发起」。

**交互：**

- 批准：生物识别 → 上传签名与决策 → 展示「处理中」或跳转结果。
- 拒绝：确认 → 上传决策 → 展示「已拒绝」并返回列表。

**错误态：** 网络失败时提示重试，并保留当前页状态以便再次提交。

---

### 3.3. 审批结果反馈

**用途：** 在用户批准或拒绝后给予明确反馈。

**形式：**

- **Toast / Snackbar：** 成功时「已批准，正在处理」；拒绝时「已拒绝」；执行失败时「支付失败：xxx」。
- **或结果页：** 单独一页展示「成功 / 已拒绝 / 支付失败」，并带「返回首页」或「查看历史」按钮。

**引导：** 反馈后自动或手动返回待审批列表或首页，便于用户处理下一条。

---

### 3.4. 添加 / 管理资产

#### 3.4.1. 添加加密货币钱包

**元素：**

- 选择链：如 Ethereum、Solana（初版可仅支持 ETH/SOL）。
- 输入或导入私钥：通过安全输入框或「粘贴」；说明私钥仅存于本机安全芯片、不会上传。
- 确认后：设备内生成或导入密钥，仅将公钥/地址同步至后端；界面展示地址与链，不展示私钥。

**状态：** 添加成功后在「资产」列表展示该钱包（地址掩码 + 链标识）。

#### 3.4.2. 添加信用卡

**元素：**

- 表单：持卡人姓名、卡号、有效期、CVV。
- 说明文案：数据经 PCI 合规金库处理，CVV 不会存储于 Aegis 服务器。
- 提交后：由 SDK 与支付处理器对接，后端仅存储金库返回的 token；界面展示卡尾号与品牌（掩码）。

**状态：** 添加成功后在「资产」列表展示该卡（如 `•••• 4242` + 品牌图标）。

#### 3.4.3. 资产列表（管理）

- 展示所有已添加的钱包与卡；支持设为默认、删除（需二次确认 + 生物识别）。
- 不展示完整卡号或私钥。

---

### 3.5. 历史 / 审计

**用途：** 所有请求的完整记录，便于用户审计与排查。

**元素：**

- 列表：每条包含时间、状态（approved / denied / failed / executed 等）、金额、收款方、来源 Agent。
- 筛选：按状态、时间范围、金额范围（可选）。
- 搜索：按收款方或描述（可选）。
- 点击条目：详情页展示原始请求内容与执行结果（如 tx_hash、payment_id、错误信息）。

**状态：** 数据只读、不可篡改；与后端同步，支持分页或「加载更多」。

---

### 3.6. 设置

**元素：**

- **通知：** 推送总开关；审批请求提醒开关。
- **默认支付方式：** 在已添加的资产中选择默认卡或钱包。
- **安全：** 生物识别开关、修改设备 PIN 说明（若适用）。
- **账户：** 登录邮箱/账号、登出。
- **关于与合规：** 版本号、隐私政策、服务条款、PCI 等合规说明入口。

## 4. 推送与深链

### 4.1. 推送文案规范

- **标题：** 固定或模板，如 `Aegis: 审批请求` / `Aegis: Approval Request`。
- **正文：** 模板化，包含金额与收款方，例如：
  - 中文：`您的 AI 助手请求向 United Airlines 支付 $149.99，请审批。`
  - 英文：`Your AI agent wants to pay $149.99 to United Airlines. Tap to review.`

推送内容不包含敏感信息（不包含完整卡号或私钥）。

### 4.2. Deep Link / Universal Link

- **点击推送：** 打开 Aegis App 并跳转至「审批详情」页。
- **URL 形式：** 如 `aegis://approve?request_id=req_abc123` 或 `https://app.aegis.com/approve?request_id=req_abc123`（Universal Link）。
- **参数：** 至少包含 `request_id`，App 据此拉取详情并展示；若未安装 App，可降级至 App Store / Play Store 或 H5 引导页。

### 4.3. 冷启动与热启动

- 冷启动：从推送打开时先展示 Splash/加载，再解析 deep link 并拉取详情，最后展示审批详情页。
- 热启动：App 已在后台时，直接解析 deep link 并刷新或打开对应审批详情，避免重复推送打开多个相同页。

## 5. 无障碍与国际化

### 5.1. 无障碍

- **关键控件：** 批准/拒绝按钮、金额、收款方需支持 VoiceOver（iOS）与 TalkBack（Android）的合理朗读顺序与描述。
- **对比度：** 主按钮与重要文字满足 WCAG AA 对比度要求。
- **焦点顺序：** 键盘/辅助设备导航时焦点顺序符合逻辑。

### 5.2. 国际化

- 所有用户可见文案使用本地化 key，至少支持：
  - 英文（en）
  - 简体中文（zh-Hans）
- 金额与日期格式随系统或应用语言与地区设置。
- 推送文案按用户语言偏好选择模板。
